# Coding Standards — Edupro SMS

## App & Module Structure

- All Edupro code lives in the custom app `edupro_sms` (snake_case app name).
  Future apps (`edupro_finance`, `edupro_attendance`, `edupro_transport`)
  follow the same pattern.
- **Never edit files under `apps/frappe` or `apps/erpnext`.** Customize only
  via: custom apps, Custom Fields, Property Setters, Client Scripts, Server
  Scripts, and fixtures.
- Organize DocTypes into logical modules within `edupro_sms` (e.g.
  `academics`, `people`, `assessments`, `reporting`) rather than one flat
  module.

## Naming Conventions

- **DocType names:** PascalCase, singular (e.g. `Academic Year`, `Student`,
  `Subject Allocation`).
- **Fieldnames:** snake_case (e.g. `academic_year`, `enrollment_date`).
- **Python files/modules:** snake_case, matching Frappe's autogenerated
  doctype folder/file naming.
- **Child tables:** named for what they represent, not the parent (e.g.
  `Marks Entry`, not `Assessment Row`).
- **Roles:** Title Case, descriptive (e.g. `Headmaster`, `Class Teacher`).

## Python

- Target the Python version bundled with the chosen Frappe release.
- Use type hints on function signatures where they clarify intent
  (especially for controller methods and shared utility functions).
- Keep methods focused and reusable — prefer several small, named methods
  over one large `validate()`/`on_submit()` doing everything inline.
- Add comments only when they explain a non-obvious *why* (a business rule,
  a workaround, a constraint from Frappe internals) — not *what* the code
  does.
- Use Frappe's own idioms (`frappe.get_doc`, `frappe.db.get_value`,
  `frappe.qb` for query builder) rather than raw SQL unless there's a
  measured performance reason, and document that reason inline when you do.

## DocTypes & Data Model

- Prefer standard DocType features (Link fields, Child Tables, Select
  fields, fetch-from) over ad-hoc Python glue.
- Use Child Tables for one-to-many data owned by a parent (e.g. marks rows
  inside an Assessment).
- Model calculated values (grades, class position) via controller methods
  or scheduled jobs, not Client Script hacks, so logic stays testable and
  server-authoritative.

## Server Scripts vs. App Code

- Server Scripts are for quick, low-risk, admin-tunable logic that
  non-developers may need to adjust (e.g. simple validation tweaks).
- Anything with real business logic, anything reused across DocTypes, or
  anything that should be code-reviewed and tested belongs in versioned app
  code (doctype controller `.py` files, `hooks.py`, `utils/`).

## Fixtures

- Export Roles, Custom Fields, Property Setters, Workflows, and Print
  Formats as fixtures (`hooks.py` `fixtures` list + `bench export-fixtures`)
  so they're version-controlled and reproducible across environments/sites.

## Workflows & Permissions

- Use the Workflow doctype for multi-step approvals (e.g. report card
  sign-off) instead of hand-rolled status fields with ad-hoc transitions.
  The canonical states/transitions are documented in
  `docs/04_Workflows.md` — implement exactly that state machine, don't
  invent a different one in code.
- Use the Role Permissions Manager for access control, matching the matrix
  in `docs/11_Roles_And_Permissions.md`. Only fall back to
  permission-query hooks (`get_permission_query_conditions`) when role-level
  permissions genuinely can't express the rule (e.g. "teacher can only see
  their own class's students").
- **Marks are locked server-side after Headmaster approval** — enforce this
  in the DocType controller (`validate`/`before_save`), not just by hiding
  the edit UI. Only System Manager can override, and any override must be
  audit-logged (see below).
- Rely on Frappe's built-in document versioning and Comments for the audit
  trail (who entered/approved/rejected marks and when) rather than building
  a parallel audit-log table, unless a specific report needs structured
  audit data Frappe's version log can't answer directly.

## Reports & PDFs

- Use Print Formats for report cards and other printed output, not
  hand-built HTML-to-PDF pipelines outside Frappe's print system.
- Use Frappe's Query Report / Script Report framework for dashboards and
  tabular reports rather than custom API endpoints feeding a bespoke UI,
  unless the MVP explicitly calls for a custom page.

## Background Jobs & Email

- Use RQ background jobs (`frappe.enqueue`) for bulk operations, especially
  bulk parent emailing — never send bulk email synchronously in a request.
- Use Frappe's Email Queue (SMTP) rather than a separate email service for
  MVP.

## Git

- Commit after each completed, tested feature (see `CLAUDE.md` daily
  workflow).
- Keep `edupro_sms` in its own Git repo, separate from the `docker/` infra
  config.

## Documentation

- Update `docs/` and `.claude/CHANGELOG.md` after completing each feature —
  don't let documentation drift from what's actually built.
