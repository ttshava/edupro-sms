# Coding Standards — Edupro SMS

## App & Module Structure

- All Edupro code lives in the custom app `edupro_sms` (snake_case app
  name). Future apps (`edupro_finance`, `edupro_attendance`,
  `edupro_transport`) follow the same pattern.
- **Never edit files under `apps/frappe`, `apps/erpnext`, or
  `apps/education`.** Customize only via: the custom app, Custom
  Fields, Property Setters, Client Scripts, Server Scripts, and
  fixtures.
- `edupro_sms` owns exactly one Frappe module, **"Edupro SMS"**
  (`edupro_sms/modules.txt`). DocTypes, Print Formats, and Reports live
  under the nested `edupro_sms/edupro_sms/` module folder; business-logic
  `.py` files that aren't tied to a specific DocType live at the outer
  package root (see the root [`README.md`](../README.md) for the full
  layout). Don't invent additional sub-modules — Frappe's tooling
  expects one module folder per entry in `modules.txt`.
- Internal imports must be package-qualified
  (`from edupro_sms.grading import ...`), never bare
  (`from grading import ...`) — bare imports only work when the app's
  own directory happens to be on `sys.path`, which breaks once the app
  is installed as a normal Python package (e.g. on Frappe Cloud).

## Naming Conventions

- **DocType names:** PascalCase, singular (e.g. `Student Fee`, `Report
  Card`).
- **Fieldnames:** snake_case (e.g. `academic_year`, `boarding_type`).
- **Python files/modules:** snake_case, matching Frappe's autogenerated
  doctype folder/file naming.
- **Child tables:** named for what they represent, not the parent (e.g.
  `Report Card Assessment Result`, not `Report Card Row`).
- **Roles:** Title Case, descriptive (e.g. `Headmaster`, `Class
  Teacher`, `Bursar`).

## Python

- Target the Python version bundled with the site's Frappe release
  (3.10+ minimum, per `pyproject.toml`).
- Use type hints on function signatures where they clarify intent —
  especially controller methods and shared utility functions.
- Keep methods focused and reusable — prefer several small, named
  methods over one large `validate()`/`on_submit()` doing everything
  inline.
- Add comments only when they explain a non-obvious *why* (a business
  rule, a workaround, a constraint from Frappe internals) — not *what*
  the code does.
- Use Frappe's own idioms (`frappe.get_doc`, `frappe.db.get_value`,
  `frappe.qb` for query building) rather than raw SQL unless there's a
  measured performance reason — and document that reason inline when
  you do.

## DocTypes & Data Model

- Prefer standard DocType features (Link fields, Child Tables, Select
  fields, fetch-from) over ad-hoc Python glue.
- Use Child Tables for one-to-many data owned by a parent (e.g. subject
  rows inside a Report Card).
- Model calculated values (grades, class position) via controller
  methods, not Client Script hacks, so logic stays testable and
  server-authoritative.

## Server Scripts vs. App Code

- Server Scripts are for quick, low-risk, admin-tunable logic that
  non-developers may need to adjust.
- Anything with real business logic, anything reused across DocTypes,
  or anything that should be code-reviewed and tested belongs in
  versioned app code (doctype controllers, `hooks.py`, business-logic
  modules).

## Fixtures

- Export Roles, Custom Fields, Workflows, Workflow States/Actions, and
  Custom DocPerms as fixtures (`hooks.py`'s `fixtures` list +
  `bench export-fixtures`) so they're version-controlled and
  reproducible across environments/sites.

## Workflows & Permissions

- Use the Workflow doctype for multi-step approvals instead of
  hand-rolled status fields. The canonical states/transitions are
  documented in `docs/04_Workflows.md` — implement exactly that state
  machine.
- Use the Role Permissions Manager for access control, matching
  `docs/11_Roles_And_Permissions.md`. Only fall back to permission-query
  hooks (`get_permission_query_conditions`/`has_permission`) when
  role-level permissions genuinely can't express the rule (e.g. "a
  teacher can only see their own assigned classes/subjects").
- **Marks are locked server-side after Headmaster approval** — enforced
  by Frappe's standard submit/docstatus locking, not just a hidden edit
  UI. Only System Manager can override, via cancel + amend.
- Rely on Frappe's built-in document versioning and Comments for the
  audit trail rather than building a parallel audit-log table.

## Reports & PDFs

- Use Print Formats for report cards and other printed output, not
  hand-built HTML-to-PDF pipelines outside Frappe's print system.
- Use Frappe's Query Report / Script Report framework for dashboards
  rather than custom API endpoints feeding a bespoke UI.

## Background Jobs & Email

- Use RQ background jobs (`frappe.enqueue`) for bulk operations,
  especially bulk parent emailing — never send bulk email synchronously
  in a request.
- Use Frappe's Email Account / Email Queue (SMTP) rather than a
  separate email service. Custom footer/signature branding lives on the
  Email Account record — see `docs/06_Email_System.md`.

## Documentation

- Update `docs/` after completing any feature or infrastructure change
  — don't let documentation drift from what's actually built or
  deployed.
- `.claude/CHANGELOG.md` and `.claude/DECISIONS.md` are historical logs
  — append to them, don't rewrite past entries.
